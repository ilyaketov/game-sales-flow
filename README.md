# SalesFlow

Сборка отчётов о **продажах** игровых ключей по площадкам **B2B, Chinaplay, GamersBase**
из Universal Report биллинга в форматы листов `Продажи Б2Б`, `продажи_Чайна`, `продажи_GB ` (с пробелом!).

## Запуск

```bash
pip install -r requirements.txt
streamlit run app.py
```

В UI загружаете файл вида `UniversalReport_DD_MM_YYYY_DD_MM_YYYY_shipped.xlsx`.
На выходе — один xlsx с тремя листами.

## Логика

| Площадка   | Фильтр биллинга          | Колонка суммы                                       | Валюта                   | Партнёр                                  |
| ---------- | ------------------------ | --------------------------------------------------- | ------------------------ | ----------------------------------------- |
| B2B        | `площадка == продажи б2б` | `Сумма позиции заказа в валюте партнера`           | `Валюта партнера`        | `Имя партнера из платформы` / `Партнёр`  |
| Chinaplay  | `площадка == Продажи Чайна` | `Сумма позиции заказа без учета комиссии`        | CNY (константа)          | `Физическое лицо Chinaplay`              |
| GamersBase | `площадка == Продажи ГБ` | `Сумма позиции заказа без учета комиссии`         | RUB (константа)          | `Физическое лицо Gamersbase`             |

Группировка: `(ID продукта, Продукт, Партнёр[, Валюта], Поставщик)`.
Цена = Сумма / Количество.

## Известные ограничения

1. **ID товаров** — берутся из биллинга. В эталонах часто использованы каталоговые ID
   (например, биллинг 19627112 ARC Raiders ≠ каталог 19192722 ARC Raiders UAH).
   Без листа «Каталог» приложение использует биллинговые ID.
2. **Chinaplay-суммы** расходятся с эталоном на ~2% из-за усреднения курсов CNY/RUB.
3. **Ручные продажи стока** через другие подразделения (исторические продажи Chinaplay-/GB-стока
   через B2B и т.д.) НЕ собираются — только апрельские транзакции из биллинга.
